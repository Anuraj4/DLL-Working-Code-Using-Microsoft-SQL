PURPOSE: Core interface and context classes for the rules engine
DESCRIPTION:
  This file defines the fundamental contracts for the rules system:
  
  - IRuleDefinition Interface:

    * Priority (int): Execution order - lower priority = executes first
    * TargetField (string): EDI field targeted (e.g., "BPR04", "CLP02", "CAS01")

    * Execute(context): Executes rule and returns determined value

    * TargetField (string): EDI field being determined
    * RawValue (string): Raw value from EOB data
    * DataModel: Complete data model with Header, Claims, ServiceLines
    * RowData (Dictionary): Row-specific context data
    * Mappings: MappingConfiguration with lookup tables
    * CrossWalkService: External cross-walk service reference
  
  This is the foundation for the entire rules engine architecture.
  Every rule implementation must implement IRuleDefinition interface.

USAGE: Referenced by all 26 other rule files in the system

FILE: RuleExecutionEngine.cs
LOCATION: Generator835/Rules/
PURPOSE: Orchestrates execution of rules in priority order
DESCRIPTION:
  Implements the Chain of Responsibility pattern for rule execution:
  
  - Constructor:
    * Takes IEnumerable<IRuleDefinition> rules
    * Automatically sorts rules by Priority (ascending)
  
  - DetermineValue(targetField, context):
    * Main entry point for field value determination
    * Executes all applicable rules in priority order
    * Returns final determined value
  
  - ResolveValue(targetField, context):
    * Extended version returning RuleResolution with audit trail
    * Tracks every rule checked and its result
    * Useful for compliance and debugging
  
  - AddRule(rule):
    * Allows dynamic rule addition at runtime
    * Re-sorts rule list by priority
    * Enables payer-specific rule overrides
  
  Process Flow:
    1. Find all rules applicable to targetField (TargetField == field OR TargetField == "*")
    2. Execute in priority order
    3. First rule returning non-null value wins
    4. If all rules return null, fallback to raw value
    5. Log all decisions at each step
  
  OPEN-CLOSED PRINCIPLE: Add new rules without modifying this class

DEPENDENCIES: IRuleDefinition, RuleResolution, Serilog

    - INFO: Each successful match with source indication

FILE: Svc01QualifierRule.cs
LOCATION: Generator835/Rules/
PURPOSE: Extracts product/service ID qualifier from procedure codes
DESCRIPTION:
  Determines SVC01-01 segment value based on code format analysis:
  
  Properties:
    - Name: "SVC01-01 Regex Qualifier Rule"
    - Priority: 10 (standard lookup priority)
    - TargetField: "SVC01-01"
  
  Extraction Strategy - Four Methods:
    
    Method 1: Prefix Pattern Matching (Regex)
      - Pattern: ^([a-zA-Z0-9]{2})[:>\.]
      - Extracts 2-char prefix followed by : > or .
      - Examples:
        * "HC:937033" → "HC"
        * "DA>5678" → "DA"
        * "J.99214" → "J"
    
    Method 2: Separator-Based Extraction
      - If contains : or >, split on that character
      - Take first non-empty part before separator
      - Examples:
        * "CPT:99214" → "CPT"
        * "HCPCS>J1100" → "HCPCS"
    
    Method 3: Dynamic Pattern Loading
      - Load "svc_qualifier_patterns" from MappingConfiguration
      - For each pattern record:
        * Check if code matches regex pattern
        * Return associated qualifier
        * Skip invalid regex patterns gracefully
      - Enables customization without code changes
    
    Method 4: Default Fallback
      - If no pattern matches, return "HC" (HCPCS)
      - "HC" is most common qualifier
  
  Supported Code Types:
    - CPT (Current Procedural Terminology)
    - HCPCS (Healthcare Common Procedure Coding System)
    - NDC (National Drug Code)
    - Revenue Codes
    - Dental Codes
    - And others (via dynamic patterns)
  
  FLEXIBILITY: Dynamic patterns from config enable customization
  ROBUSTNESS: Multiple fallback strategies ensure qualification

DEPENDENCIES: IRuleDefinition, System.Text.RegularExpressions

FILE: Svc01ProcedureCodeRule.cs
LOCATION: Generator835/Rules/
PURPOSE: Sanitizes procedure codes by removing prefixes and suffixes
DESCRIPTION:
  Cleans procedure codes to X12-compliant format:
  
  Properties:
    - Name: "SVC01-02 Procedure Code Sanitizer"
    - Priority: 10 (standard lookup priority)
    - TargetField: "SVC01-02"
  
  Sanitization Process - Three Steps:
    
    Step 1: Remove Prefix Qualifiers
      - Pattern: ^(?:[a-zA-Z0-9]{2}[:>\.]\s*)+(.*)$
      - Removes 2-char qualifier followed by : > or .
      - Examples:
        * "HC:937033" → "937033"
        * "DA>5678" → "5678"
        * "J.99214" → "99214"
      - Trims whitespace from result
    
    Step 2: Remove Suffix Modifiers (with separators)
      - If contains : > or .
      - Split on these separators
      - Keep first part (base code)
      - Examples:
        * "99214:25" → "99214"
        * "J1100>LR" → "J1100"
        * "937033.25" → "937033"
    
    Step 3: Handle Concatenated Modifiers
      - If no separators found AND code length is 7 or 9 chars
      - Extract first 5 characters as base code
      - Removes seamlessly concatenated modifiers
      - Examples:
        * "9921425" (7 chars) → "99214" (CPT + 2-digit modifier)
        * "J110025LR" (9 chars) → "J1100" (NDC + modifiers)
  
  Return Value:
    - If all steps fail: Return original code
    - Ensures non-null return for valid codes
    - Returns empty string for empty input
  
  CRITICAL: Ensures X12 format compliance
  ORDER IMPORTANT: Steps executed in specific order to handle all cases

DEPENDENCIES: IRuleDefinition, System, System.Text.RegularExpressions

DESCRIPTION:

      PR_Total = Allowed Amount - Paid Amount
      Used to determine patient copay/deductible/coinsurance
    
    Contractual Obligation (CO):
      CO_Total = Charge Amount - Allowed Amount
      Write-off amount per healthcare contracts
    
    Balance Check:
      Gap = Charge - Paid - Sum(Adjustments) MUST EQUAL 0
      Ensures all amounts reconcile perfectly
  
  Patient Responsibility (PR) Logic - Four Rules:
    
    Rule 1: Trust Excel Breakdown (Highest Priority)
      - If Excel contains copay/deductible/coinsurance values
      - ALWAYS use Excel values for PR-3/PR-1/PR-2
      - Most reliable source of patient responsibility
    
    Rule 2: Calculate from Amounts
      - If no Excel breakdown:
        * Calculate PR = Allowed - Paid
        * Check if PR matches CLP05 perfectly → Use PR-3
        * Check PR ± Sequestration matches CLP05 → Use PR-3
        * If no exact match → Fallback to PR-3
    
    Rule 3: Qualify PR Results
      - PR-3: Standard patient responsibility (copay/coinsurance)
      - PR-1: Deductible
      - PR-2: Coinsurance
    
    Rule 4: Fallback Strategy
      - If all else fails, return PR-3 with calculated amount
  
  Contractual Obligation (CO) Logic:
    
    Phase 1: Preserve Excel CO/OA Values
      - Non-special CO and OA adjustments from Excel preserved
    
    Phase 2: Handle Sequestration
      - Excel sequestration column → CO-253
      - Other insurance column → OA-23
    
    Phase 3: Balance Check
      - If Gap == 0 → Balanced, done
      - If Gap > 0 → Check for sequestration first, remainder to CO-45
      - If Gap < 0 → Reduce or remove CO-45
    
    Phase 4: Sequestration Detection (Gap > 0 only)
      
      Medicare Rules:
        - ALWAYS: CO-253 = exactly 2% of Allowed Amount
        - Non-negotiable, regulatory requirement
      
      Non-Medicare Rules:
        - If GAP ≈ 2% of Allowed (within $0.01)
          * Set CO-253 = GAP
          * Assume sequestration
        - If GAP not near 2%
          * Skip CO-253
          * Use CO-45 only
      
      Guard Condition:
        - If CO-253 already placed in Phase 3B
        - Skip detection entirely (prevent duplication)
  
  Sequestration (CO-253) Significance:
    - Medicare uses for regulatory compliance
    - Hospital tax/sequestration holdback
    - Cannot be sent to patient (provider responsibility)
    - Must be precisely calculated per regulations
  
  CO-45 (Gap Fill) Purpose:

    - Fills unaccounted financial gaps
    - Last resort if no specific reason code applies

SECTION 10: SYSTEM ARCHITECTURE OVERVIEW
================================================================================

DESIGN PATTERN: Chain of Responsibility
  - Rules form a chain where first match wins
  - Each rule can pass to next rule by returning null
  - Order matters: defined by Priority value
  
PRINCIPLES:
  - Open-Closed: Add rules without modifying engine
  - Single Responsibility: Each rule handles one concern
  - Dependency Injection: Context passed to all rules
  - Configurability: Many rules data-driven from CSV tables

EXECUTION FLOW:
  1. Generate field value needed
  2. Create RuleExecutionContext with all data
  3. Call RuleExecutionEngine.DetermineValue(field, context)
  4. Engine finds applicable rules for field
  5. Execute rules in priority order
  6. First non-null return is final value
  7. If all null, use raw value as fallback
  8. Return RuleResolution with full audit trail

PRIORITY ORGANIZATION:
  Priority 5: High-impact domain rules (CARC/CAGC, balancing, etc.)
  Priority 8: Specialized pattern matching (claim filling indicators)
  Priority 10: Standard CSV and Trn03 lookups
  Priority 20: Cross-walk and conditional logic
  Priority 30-50: Calculated segment values
  Priority 80: Payment settings (late lookup)
  Priority 100: Field-specific defaults
  Priority 150: Absolute catch-all default

KEY RULES BY CATEGORY:

  Financial:
    - MathBalancingRule: Ensures charge/paid/adjustment balancing
    - TransactionHandlingRule: BPR calculations
    - PaymentSettingRule: Payment defaults

  Code Mappings:
    - CarcCagcRule: Healthcare-specific CARC→GAGC mapping
    - ClaimFillingIndicatorRule: Claim type→indicator mapping
    - CsvLookupRule: Generic CSV table lookups
    - Svc01QualifierRule: Procedure code qualifier detection

  Identification:
    - PayerIdentificationRule: Payer (N103/N104)
    - PayeeIdentificationRule: Provider (N103/N104, NPI priority)
    - PatientIdentificationRule: Patient/member (NM108/NM109)
    - RenderingProviderRule: Service provider (NM108/NM109)
    - Trn03Rule: Transaction payer ID

  Communication:
    - PayerCommunicationRule: Auto-detect email/URL/phone
    - ContactInfoRule: Contact name and number defaults

  Validation:
    - AdjustmentSanitizationRule: Valid CAS01 codes only
    - RemarkCodeQualifierRule: NCPDP vs standard codes

  Structural:
    - ServiceDateRule: Single date vs date range
    - RegexTransformRule: Pattern-based extraction
    - ConditionalLogicRule: If/then logic
    - FallbackDefaultRule: Last-resort defaults
    - CrossWalkRule: External service lookups

DATA DEPENDENCIES:
  All rules depend on rich RuleExecutionContext containing:
  - TargetField: What field to determine
  - RawValue: Starting value from EOB
  - DataModel: Full claim/header/service-line data
  - RowData: Row-level context
  - Mappings: Configuration and lookup tables
  - CrossWalkService: External code services

EXTENSIBILITY:
  To add new rule:
  1. Implement IRuleDefinition interface
  2. Set Name, Priority, TargetField
  3. Implement CanExecute(context) predicate
  4. Implement Execute(context) logic
  5. Register in RuleRegistry.CreateEngine()
  6. Set appropriate priority based on rule type
  
  No changes needed to RuleExecutionEngine!

LOGGING & AUDITING:
  - All decisions logged via Serilog
  - RuleResolution provides complete audit trail
  - Can replay rule decisions for compliance review
  - Essential for healthcare audit requirements

TESTING STRATEGY:
  - Each rule can be tested independently
  - Mock RuleExecutionContext for testing
  - No need to test engine for rule changes
  - Configuration-driven rules testable via CSV mock data
